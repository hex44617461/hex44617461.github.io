# 도커|docker 컨테이너화 도구

작성 코드가 모든 컴퓨터에서 동일 환경으로 실행될 수 있도록 패키징하고 관리하는 컨테이너화 도구

## 도커-개념-역사

### 역사-탄생

1. 닷클라우드에서 간편하게 사용할 수 있는 내부 격리 시스템 개발:
	- 프랑스 개발자 '솔로몬 하이크\|Solomon Hykes'와 그의 동료는 '닷클라우드\|dotCloud' 창업
	- 닷클라우드는 AWS·Heroku 같이 PaaS 서비스를 제공함
	- 다양한 언어·프레임워크로 구성된 고객의 개발 환경을 안전하고 효율적으로 관리 해야했음
	- 가상머신(VM)은 너무 무겁고 느려서 비용 감당이 안됐고, 리눅스컨테이너(LXC)는 일반 개발자가 쓰기 복잡함
	- 이런 배경으로, VM보다 가볍고 LXC보다 편리한, 파이썬 기반의 간편 격리 시스템을 고안함
2. 도커\|Docker 공개:
	- 닷클라우드 주력 비즈니스가 위기에 빠짐
	- 솔로몬 하이크는 PyCon에서 내부 격리 시스템을 Docker 라는 이름으로 오픈소스로 공개함
	- 전세계 개발자가 겪고 있는 '내 컴퓨터에서는 되는데 서버에서 안 됨' 이라는 문제를 해결하는 도구로 평가됨
	- 개발자 사이에서 큰 호응을 받았고, 닷클라우드는 도커\|Docker Inc로 이름을 바꾸게 됨
3. 도커 서비스 운영: 
	- 도커 데스크탑\|Docker Desktop, 도커 허브\|Docker Hub 서비스 제공
	- 도커 데스크탑은 도커를 쉽게 쓸 수 있는 GUI 프로그램, 일정 규모 이상 조직에서는 구독료 청구
	- 도커 허브는 도커 이미지를 저장·공유하는 원격 저장소, 비공개로 일정 공간 이상 사용하려면 구독료 청구

### 역사-경쟁

1. 도커 문제점 부상:
	- 도커의 유료화 정책으로 비용 부담이 발생하기 시작함
	- 도커 데몬이 백그라운드에서 Root 권한으로 실행 돼서, 탈취 당하면 Host 컴퓨터 장악 당하는 취약점 존재
2. Podman 등장:
	- 레드햇\|RedHat 주도로 개발된 Docker 대체 도구
	- `docker` 대신 `podman` 쓰면 명령 그대로 호환
	- 도커 데몬 같은 것 없이, Host에서 리눅스 커널을 직접 소통(리눅스 커널 VM 필요)
	- 포드맨 데스크탑\|Podman Desktop이 오픈소스 공개됨

## 도커-개념-구조

### 구조-Host

Docker Client와 Docker Daemon이 실행되고 있는 컴퓨터

### 구조-Docker Client

Host 터미널에서 내린 `docker build`, `docker run` 같은 명령을 REST API로 Docker Daemon 에게 전달

### 구조-Docker Daemon

Docker Client에서 받은 명령을 바탕으로, Host에 설치된 리눅스 커널을 이용해서, Docker Image와 Docker Container 관리:

`docker build` 명령을 받으면, Host 백그라운드에서 Dockerfile 1줄씩 실행하면서 이미지 빌드:

- 이미 있는 Image는 새로 만들지 않고 참조(캐시)만 함

### 구조-Docker Registry

완성된 Docker Image를 전세계와 공유하는 일종의 원격저장소([Docker Hub](https://hub.docker.com/)):

- `docker pull`: Docker Hub에서 지정한 Image 가져옴
- `docker push`: Docker Hub에서 지정한 Image 내보냄

## 도커-개념-Image

Docker Image는 컨테이너 실행을 위한 파일

### Image-조회

`docker image ls`:

- `docker images`도 동일 작동

### Image-생성

`docker pull 이미지:태그명`:

- dockerhub ← 여기서 다운로드 받아오는것
- 태그 안쓰면 lastest 이거 최신 이미지로 다운받아짐

### Image-삭제

`docker image rm [아이디(일부값만 써도 됌)]`:

- 컨테이너에서 쓰고 있으면 삭제 못함
- `docker image rm$(docker images -q)`: 안쓰고 있는 이미지 삭제
- `docker image rm -f$(docker images -q)`: 중단된 컨테이너에서 쓰고 있는 이미지 삭제

## 도커-개념-Container

Docker Container는 도커 이미지를 실행하고 있는 프로세스:

1. 디스크 저장공간, 각 컨테이너마다 각자의 저장 공간을 가지고 있음
2. 각 컨테이너마다 고유의 네트워크를 가지고 있음, 컨테이너는 각 IP를 따로 가지고 있음

### Container-조회

`docker ps [옵션]`:

- [옵션]:
	- -a: 모든 컨테이너를 출력
	- -q: 컨테이너 ID만 출력하는 옵션
	- -qa: 모든 컨테이너의 ID만 출력
- 출력:
	- STATUS:Created = 생성됨
	- STATUS:Up = 실행중

### Container-생성

`docker create [이미지]`

### Container-실행

`docker start [컨테이너 아이디(일부값만 써도 됌)]:

- 내 로컬 환경에 다운받은 이미지가 없으면, 그냥 허브에서 다운 받아서 실행함

### Container-생성·실행

`docker run [옵션] [이미지명]`:

- [옵션]:
	- -d: 백그라운드 실행
	- -p: 포트 지정 ex) `docker run -d -p 4000:80`

### Container-중지

`docker stop [컨테이너 아이디]`:

- `docker stop [컨테이너 아이디] [컨테이너 아이디]` 이런식으로 여러 컨테이너 중단 가능

`docker kill [컨테이너 아이디]`:

- 강제종료
- 다른 쓰고 있어서 못한다는것도 그냥 종료해버림

### Container-삭제

`docker rm [컨테이너 아이디]`:

- `docker rm $(docker ps -qa)` 중지 컨테이너 모두 삭제

### Container-로그

`docker log [옵션] [컨테이너 아이디]`:

- [옵션]:
	- -f: 로그 실시간 생성
	- -tail n: 끝부분 n개 출력

### Container-접속

`docker exec [옵션] [컨테이너 아이디] [사용 도구]`:

- [옵션]:
	- -it: 접속 ex) `docker exec -it 433 bash`

## 도커-파일-Dockerfile

도커 이미지 빌드를 지도하는 파일:

- `FROM`, `CMD` 필수 절과 다른 절을 조합하여 작성
- 도커 클라이언트는 Dockerfile을 도커 데몬으로 전달
- 도커 데몬은 한 줄씩 실행하고 `FROM`, `COPY`, `ADD`, `RUN` 이 4가지에 해당하는 키워드와 만났을 때, 레이어를 1장 추가로 쌓아서 이미지 빌드
- 기존에 만든 레이어가 있다면, 빌드하지 않고 참조 함(캐싱)
	- 이미 빌드된 이미지를 참조할 때, 해당 코드가 변경 됐다면?, 레이어를 새로 만듬 (캐시가 깨졌다고 표현함)
	- 3번 레이어가 바뀌었다면 4번 레이어도 동일하다고 보장 못하니까, 레이어를 또 새로 만듬
	- 자주 바뀌는 소스코드는 최하단으로 조정하는게 좋음
- 빌드된 이미지는 수정 불가, 수정하려면 새로 빌드해야함

```dockerfile
FROM node:18          # 1. 베이스 이미지 (거의 안 바뀜 ➡️ 캐시 ⭕)
WORKDIR /app          # 2. 작업 폴더 지정 (거의 안 바뀜 ➡️ 캐시 ⭕)

# 3. 패키지 명세서만 '먼저' 복사하고 설치 (가끔 바뀜)
COPY package.json .   # COPY 2개로 나눠서 잘 안바뀔꺼 먼저 씀
RUN npm install       # ➡️ 내 소스코드를 고쳐도 패키지가 안 바뀌었으면 여기도 캐시 ⭕ (초고속 패스)

# 4. 실제 내 소스코드 전체 복사 (맨날 바뀜)
COPY . .              # ➡️ 내가 코드 고치면 딱 '여기서부터'만 캐시가 깨짐!

CMD ["node", "app.js"]
```

```Dockerfile
# 1. 베이스 이미지 선택
FROM python:3.12-slim

# 2. 작업 디렉토리 설정
WORKDIR /app

# 3. 코드 복사
COPY main.py .

# 4. 컨테이너 시작 시 실행할 명령
CMD ["python", "main.py"]
```

```
### 1. FROM (필수)
`FROM 베이스_이미지:태그`
- 빌드의 시작점. 어떤 OS나 런타임 환경 위에서 코드를 올릴지 결정함 (예: `FROM node:18`)
- **주의:** `FORM`으로 쓰면 빌드 에러 남.

### 2. WORKDIR (선택이지만 필수 권장)
`WORKDIR /프로젝트_경로`
- 컨테이너 내부에서 명령어를 실행할 '기본 작업 디렉토리'를 지정함 (리눅스의 `cd` 명령과 같음)
- 이거 안 쓰면 루트(`/`) 폴더에 파일이 마구잡이로 복사되어 환경이 꼬임.

### 3. COPY (필수 권장)
`COPY 내_컴퓨터_경로 컨테이너_경로`
- 호스트 컴퓨터에 있는 내 작성 코드를 컨테이너 내부로 복사해 넣는 명령
- 보통 `COPY . .` 형태로 현재 폴더의 모든 소스코드를 싹 다 집어넣을 때 씀.

### 4. CMD (필수)
`CMD ["실행명령", "인자"]`
- 컨테이너가 `docker run`으로 최종 실행될 때 자동으로 켜질 프로그램을 지정함 (예: `CMD ["node", "app.js"]`)
- Dockerfile 안에서 최종적으로 딱 한 번만 유효함.
```

### Dockerfile-RUN

## 도커-파일-.dockerignore

Docker 이미지 빌드에서 특정 파일·폴더를 Docker 데몬으로 전송 제외 할 것을 명시하는 파일

제외하면 여러 장점이 있음:

1. 데이터를 전송하는 시간 단축 → 빌드 속도 개선
2. 불필요한 파일이 이미지 레이어에서 제외 → 최종 이미지 용량 개선
3. 민감 데이터 제외 → 보안사고 예방

### docker compose bulid

docker-compose.yml 또는 Dockerfile 참고해서, 이미지만 생성

### docker compose up --build

docker-compose.yml 또는 Dockerfile 참고해서, 이미지 생성(이미 있다면 덮어쓰기)하고 컨테이너 실행

### docker compose up -d

docker-compose.yml 또는 Dockerfile 참고해서, 백그라운드에서 이미지 생성(이미 있다면 그거 참고)하고 컨테이너 실행

### docker compose down

실행되고 있는 컨테이너 중단하고 삭제, 이미지는 보존

6. Docker Compose (도커 컴포즈)
여러 컨테이너를 하나로 묶어서 관리할 때 사용 (docker-compose.yml 참고).

docker compose build : 이미지만 생성함.

docker compose up --build : 이미지 생성(이미 있으면 덮어쓰기)하고 컨테이너까지 실행함.

docker compose up -d : 백그라운드에서 이미지 생성(있으면 그거 참고)하고 컨테이너 실행함.

docker compose down : 실행 중인 컨테이너 중단하고 삭제함. (이미지는 보존됨)
