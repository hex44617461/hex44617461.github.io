# TCREI

# TCREI(프레임워크)

TCREI는 LLM에서 고품질 결과를 얻기 위해, 구글에서 제안한 5단 구조 프레임워크

## TCREI-요소

### 요소-Task

어떤 작업을 요청하는지 구체적이고 명확하게 정의:

- AI가 작업에서 어떤 인물 역할을 하는지 명시
- 출력은 어떤 형식으로 나와야 하는지 명시
- 작업 수행 단계와 각 단계 정의하고, 나눠서 지시

```
[역할 명시]
- 당신은 성능보다 유지 보수를 더 우선시하며, 문제점을 세심히 파악하고 구체적이고 자세하게 피드백하는 10년차 백엔드 개발자입니다

[작업 명시]
- 앞으로 당신에게 제시되는 코드에 대하여, 유지 보수 용이성 위주로 피드백하세요

[출력 명시]
- 피드백 시행할 때, 의견과 개선된 코드 완성본을 마크다운 형식으로 제시해야하며, 단계별 나눠 설명하세요
```

### 요소-Context

작업과 관련된 배경지식을 충분히 묘사:

- 작업을 요청하는 배경
- 작업 진행 프로세스 또는 적용 기술
- 제약조건 또는 예외 처리 방법

```
[배경지식 묘사]
- 리뷰할 코드는 사내 레거시 결제 시스템의 일부로, Java 17 및 Spring Boot 환경에서 작성되었습니다
- 현재 결제 모듈에서 예외 처리가 미흡하여 DB 롤백이 간헐적으로 실패하는 이슈가 발생하고 있습니다
- 외부 API 연동 지연으로 인한 타임아웃 및 예외 처리를 가장 중요하게 고려해야 합니다
```

### 요소-Reference

작업에 참고할 수 있는 자료 첨부:

- 요청하는 작업과 유사한 구조의 예시 제시
- 일부가 아닌 완전한 예시를 제시해야 추측 감소
- 과도하게 복잡한 예시는 지양
- 나쁜 예시는 사전에 제거

```
[참고자료 제시]
- 코드 개선 시, 아래의 사내 표준 예외 처리 컨벤션을 참고하여 작성해 주세요
- 좋은 예시:
  @Transactional
  public void process(Long id) {
      try {
          // 비즈니스 로직
      } catch (SpecificException e) {
          log.error("Failed to process id: {}", id, e);
          throw new BusinessException(ErrorCode.PROCESS_FAILED, e);
      }
  }
- 무분별한 Exception 상위 클래스 catch나 로그 누락 패턴은 지양해 주세요
```

### 요소-Evaluate

출력에 대한 명확한 평가 기준 정의:

- 요청에 부합 여부를 판단할 수 있는 평가 기준 제시
- 전·후 작업과 유기적 연결을 유도하는 평가 기준 사용

```
[기준 정의]
- 수정된 코드가 기존 비즈니스 로직의 흐름을 깨뜨리지 않는지 확인하세요
- 각 메서드가 단일 책임 원칙(SRP)을 준수하도록 적절히 분리되었는지 점검하세요
- 피드백 내용에 '왜 이렇게 수정해야 하는지'에 대한 명확하고 논리적인 이유가 포함되어 있는지 확인하세요
```

### 요소-Iterate

 구체적인 지적과 명확한 개선 요청:

 - 이전 단계의 내용과 부합하는 결과가 나왔는지 확인
 - 특히, 보안과 유지·보수 가능한지 확인
 - 지적하는 부분과 명확한 개선 방향과 방법 제시

```
[개선 요청]
- 기능 정확성: 피드백해 주신 코드에서 결제 취소 시의 예외 처리 로직이 누락된 것 같습니다. 이 부분을 추가해 주세요
- 코드 품질: if-else 문이 너무 깊어 가독성이 떨어지므로, Early Return 패턴을 적용해서 다시 작성해 주세요
- 유지보수성: 예외 처리 로직을 별도의 private 메서드로 분리하여 코드 응집도를 높여주세요
```
